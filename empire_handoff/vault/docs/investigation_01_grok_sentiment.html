2. Grok API x_search 技術仕様

2.1 API概要

項目内容

認証Bearer Token (XAI_API_KEY)
ツールx_search — X(Twitter)リアルタイム検索
推奨モデルgrok-4-1-fast-non-reasoning (最速・最安)
コンテキスト窓2,000,000トークン（業界最大）

2.2 x_search パラメータ

パラメータ型説明活用法
allowed_x_handlesArray (max 10)特定ユーザーの投稿のみ検索インフルエンサー監視
excluded_x_handlesArray (max 10)特定ユーザーを除外ボットアカウント除外
from_dateISO8601検索開始日過去1-24時間指定
to_dateISO8601検索終了日現在まで
enable_image_understandingBoolean画像解析有効化チャートスクショ解析

3. 3つのユースケース設計

3.1 UC-1: ダマシ防止（Damashi Guard）

コンセプト

LevBurn等のBotがシグナルを出した時、エントリー前にGrokでX投稿を検索。 ファンダ起因の一時的な価格急変と判定されたらエントリーをブロックする。 さらに related_symbols で類似銘柄も連鎖ブロック。

LevBurn: 「TRUMP_USDTにFR異常検知、ロングシグナル」
Grok x_search: 「$TRUMP crypto」で過去数時間のX投稿を検索
Grok判定: 「大統領令ニュースで急騰。is_fundamental=true, damashi_risk=high, related=[MELANIA, MAGA]」
BLOCK: TRUMP + MELANIA + MAGA のエントリーを禁止

判定ロジック

条件リスクスコア説明
is_fundamental = true+3ニュース・発表が存在
damashi_risk = high+2一時的スパイクの可能性大
damashi_risk = medium+1不明瞭なインパクト
post_volume = high+1X投稿量が異常増加
|score| > 0.7+1極端なセンチメント
related_symbols >= 3+1セクター連動（ミーム全体等）
合計アクション
≥ 4BLOCKエントリー禁止
2-3CAUTIONレバ半減 or サイズ縮小
0-1OK通常通り

昨日のTRUMP相場に適用した場合

シミュレーション結果: risk_score = 8 → BLOCK

is_fundamental(+3) + damashi_risk:high(+2) + post_volume:high(+1) + score:0.85(+1) + related:4銘柄(+1) = 8
TRUMP/MELANIA/MAGAへのエントリーがブロックされ、-54.9%の損失を回避できた。

3.2 UC-2: ファンダ飛び乗り（Fundamental Rider）

コンセプト

急騰を検知した銘柄について、Grokでファンダ要因を調査。 持続性のある本物のカタリスト（取引所上場、提携発表、規制変更等）であれば、 初動～中盤でトレンドフォローのロングエントリー。

Tier1スキャン: 「XYZ_USDTが5分で+8%急騰、出来高5倍」
Grok x_search: 「$XYZ crypto」+ 「XYZ listing announcement」で検索
Grok判定: 「Binance上場発表。catalyst_type=listing, sustainability=high, time_since_news=15min」
RIDE: ロングエントリー。TP幅広め、トレーリングストップ

カタリスト分類と対応

カタリスト種別持続性対応例
取引所上場 高 (数時間～数日) ロング飛び乗り Binance/Coinbase listing
大手提携・統合 高 ロング飛び乗り Visa提携、L2ローンチ
規制好材料 高 ロング飛び乗り ETF承認、合法化
政治家発言 中 (数時間) 慎重ロング or パス 大統領令、議員支持
インフルエンサー煽り 低 (数分～1時間) パス or ショート準備 Elon tweet、KOL推奨
ミームバイラル 低 ブロック TikTokトレンド

3.3 UC-3: 天井ショート（Ceiling Short）

コンセプト

ファンダポンプ後にX上のセンチメントがピーク→反転し始めたタイミングを検知。 過熱度が最大で反転兆候が出たらショートエントリー。

監視: 過去2時間で+20%以上急騰した銘柄をリストアップ
Grok x_search (2段階): ① 1時間前のセンチメント vs ② 直近のセンチメント
Grok判定: 「1h前 score=0.9 → 現在 score=0.5。ピーク通過、利確投稿増加中」
SHORT: ショートエントリー。タイトSL（反発リスク対応）

天井検知シグナル

シグナルX上の兆候重み
センチメント反転score低下: 0.9 → 0.5（30%以上の低下）+3
利確報告増加「sold」「profit taken」「cashed out」の投稿増+2
FUD出現批判・疑念の投稿が増え始める+2
投稿量ピーク通過1h前 > 現在の投稿量+1
インフルエンサー離脱主要KOLが言及を停止+1
合計 ≥ 5→ ショートエントリー推奨

2段階Grokクエリ設計

Query 1: 1時間前のスナップショット

x_search:
  from_date: T-2h
  to_date: T-1h
prompt: sentiment score for $XYZ

Query 2: 直近のスナップショット

x_search:
  from_date: T-1h
  to_date: now
prompt: sentiment score +
  profit-taking signals for $XYZ

2回のAPIコール（$0.012）でセンチメント変化率を計算。score_delta = query2.score - query1.score。
score_delta ≤ -0.3 かつ query1.score ≥ 0.7 → 天井通過シグナル。

4.5 キャッシュ戦略

重要: LevBurn-Secは5秒間隔スキャン

毎回Grok APIを呼ぶとコストとレイテンシが問題。銘柄ごとに5分間キャッシュする。

class GrokSentimentCache:
    def __init__(self, ttl_seconds=300):
        self._cache = {}   # {symbol: {data, timestamp}}
        self._blocked = {}  # {symbol: {until, reason}}  ← ブロック状態

    async def check(self, symbol, direction) -> GrokVerdict:
        # キャッシュヒットなら即返す
        if symbol in self._cache and not expired:
            return self._cache[symbol]

        # API呼び出し
        result = await self._query_grok(symbol)
        self._cache[symbol] = result

        # BLOCK判定ならブロックリストに追加（30分）
        if result.action == "BLOCK":
            self._blocked[symbol] = now + 30min
            for related in result.related_symbols:
                self._blocked[related] = now + 30min

        return result

7. Iron Rule適合性

CLAUDE.md Iron Rules との整合性
Iron Rule内容適合性説明
#1 AIはTier 3コメンタリーのみ。価格判定・計算にAIを関与させない 要注意 GrokはI「価格を予測」するのではなく「ファンダ情報の有無」を判定する。
価格判定そのものではないが、エントリー可否に影響するためゲート（門番）として位置づける。
最終的な価格判定（TP/SL設定）はCCXTデータに基づくBotロジックが行う。
→ 「ファンダ情報フィルター」として位置づければIron Rule #1に抵触しない
#2 数値データはCCXT経由の生値のみをSource of Truth 適合 Grokからの数値（score等）は取引判断の補助情報であり、価格・出来高の代替ではない。 CCXT由来のデータ構造は変更しない。
#3 データ欠損はNone。推測値で埋めない 適合 GrokがAPI障害等で応答しない場合、デフォルト=OK（通過）。推測値は使わない。
#4 Tier 1→2→3のファンネル構造を厳守 適合 Grokゲートは既存ファンネルの後段フィルターとして追加。 ファンネル構造は変更しない。
Iron Rule #1 との共存方針

Grokの役割を「情報フィルター」に限定する:

  • Grokは「ニュースが存在するか」「X投稿量が異常か」を事実として検知する
  • 「価格がいくらになるか」「このトレードは儲かるか」は判断しない
  • 最終エントリー判断は既存Botロジック（CCXT数値ベース）が行う
  • Grokは「通す/止める」のバイナリ判定のみ

8. リスクと制約

リスク影響度対策
Grok API障害・レイテンシスパイク 中 タイムアウト3秒設定。障害時はフォールスルー（OK判定）で通常運転継続
誤判定: 本物のチャンスをBLOCK 中 risk_score閾値を調整可能。ログ記録で誤判定率を追跡・改善
Xの情報操作（フェイクニュース） 高 confidence閾値導入。信頼度低い情報では判定しない。複数ソース要求
APIコスト超過 低 日次上限設定（例: 500回/日=$3）。キャッシュで実質50-70%削減
x_searchがパブリック投稿のみ 低 クリプト情報の大半はパブリック。DM/プライベート情報は元々アクセス不可
xAI API仕様変更（旧API廃止済み） 中 現在のResponses API + Toolsは最新方式。SDKバージョン追従
Iron Rule #1への抵触懸念 中 「情報フィルター」として位置づけ。価格予測には関与しない設計
